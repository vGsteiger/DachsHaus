# DachsHaus Security Checklist

This checklist helps ensure your DachsHaus installation is secure, especially when moving from development to production.

## 🔐 Pre-Production Security Checklist

### Authentication & Authorization

- [ ] **Change default admin password immediately**
  - Default: `admin@dachshaus.local` / `admin123`
  - Change on first login through the storefront

- [ ] **Rotate all generated secrets**
  ```bash
  # Generate new HMAC secret
  openssl rand -base64 32 | tr -d "=+/" | cut -c1-32
  # Update .env: GATEWAY_HMAC_SECRET
  ```

- [ ] **Regenerate JWT keys for production**
  ```bash
  openssl genrsa -out keys/auth-private-prod.pem 4096
  openssl rsa -in keys/auth-private-prod.pem -pubout -out keys/auth-public-prod.pem
  chmod 600 keys/auth-private-prod.pem
  ```

- [ ] **Configure JWT token expiration**
  - Access tokens: 15 minutes (default) or shorter for high-security
  - Refresh tokens: 7 days (default) or shorter
  - Update in Auth Service configuration

- [ ] **Implement rate limiting**
  - Login attempts: 5 per 15 minutes (already configured)
  - API requests: Configure at gateway level
  - Consider adding IP-based rate limiting

- [ ] **Review RBAC permissions**
  - Audit who has admin access
  - Create custom roles if needed
  - Test permission boundaries

### Network Security

- [ ] **Enable SSL/TLS certificates**
  ```bash
  # For production, use Let's Encrypt or your CA
  # Update docker-compose.prod.yml with SSL config
  ```

- [ ] **Configure firewall rules**
  - Only expose ports 80 (HTTP) and 443 (HTTPS)
  - Keep database ports (5432, 6379, 9092) internal only
  - Use security groups in cloud deployments

- [ ] **Implement network segmentation**
  - Already configured: public/internal Docker networks
  - Verify services can't access what they shouldn't
  - Review Kubernetes network policies (if using K8s)

- [ ] **Enable DDoS protection**
  - Use cloud provider DDoS protection (e.g., GCP Cloud Armor)
  - Configure rate limiting at load balancer
  - Set up request throttling

- [ ] **Secure service-to-service communication**
  - HMAC signatures already configured between gateway and services
  - In Kubernetes: Enable Istio mTLS (configured in k8s manifests)
  - Verify signature validation is working

### Data Security

- [ ] **Encrypt data at rest**
  - PostgreSQL: Enable transparent data encryption
  - Redis: Enable RDB encryption (already configured with password)
  - Kafka: Configure encryption at rest
  - Cloud: Use CMEK (Customer-Managed Encryption Keys)

- [ ] **Encrypt data in transit**
  - Enable TLS for PostgreSQL connections
  - Enable TLS for Redis connections
  - Enable TLS for Kafka (production recommendation)
  - All external APIs should use HTTPS only

- [ ] **Secure database connections**
  - Use strong passwords (generated by installer)
  - Rotate database passwords regularly
  - Use SSL certificates for DB connections
  - Restrict database access to application network only

- [ ] **Implement data retention policies**
  - Define retention period for user data
  - Implement automatic data deletion (GDPR compliance)
  - Configure log retention (7-30 days recommended)
  - Set up backup retention policy

### Secrets Management

- [ ] **Never commit secrets to git**
  - Verify `.gitignore` includes `.env`, `CREDENTIALS.txt`, `keys/`
  - Audit git history for leaked secrets
  - Use git-secrets or similar tools

- [ ] **Use secrets management service**
  - Production: GCP Secret Manager (Terraform configured)
  - Kubernetes: External Secrets Operator (configured in k8s manifests)
  - Rotate secrets automatically

- [ ] **Secure credential storage**
  ```bash
  # Verify file permissions
  chmod 600 .env
  chmod 600 CREDENTIALS.txt
  chmod 600 keys/*.pem
  ```

- [ ] **Implement secret rotation**
  - Database passwords: Every 90 days
  - HMAC secrets: Every 90 days
  - JWT keys: Every 180 days
  - Document rotation procedure

### Application Security

- [ ] **Keep dependencies up to date**
  ```bash
  # Check for vulnerabilities
  cd services && ./gradlew dependencyCheckAnalyze
  pnpm audit

  # Update dependencies
  pnpm update --latest
  ```

- [ ] **Enable security headers**
  - Content-Security-Policy
  - X-Frame-Options: DENY
  - X-Content-Type-Options: nosniff
  - Strict-Transport-Security (HSTS)
  - Already configured in gateway and storefront

- [ ] **Implement input validation**
  - GraphQL schema validation (already configured)
  - SQL injection prevention (using prepared statements)
  - XSS prevention (React sanitization + CSP)
  - Command injection prevention

- [ ] **Configure CORS properly**
  - Restrict origins to your domains only
  - Update gateway CORS configuration
  - Don't use wildcard (*) in production

- [ ] **Implement audit logging**
  - Log all authentication attempts
  - Log all admin actions
  - Log all data modifications
  - Retain logs for compliance

### Infrastructure Security

- [ ] **Harden Docker containers**
  - Run as non-root user (already configured)
  - Use minimal base images
  - Scan images for vulnerabilities
  - Remove unnecessary packages

- [ ] **Configure resource limits**
  ```yaml
  # In docker-compose.prod.yml
  deploy:
    resources:
      limits:
        cpus: '2'
        memory: 2G
      reservations:
        cpus: '1'
        memory: 1G
  ```

- [ ] **Implement backup strategy**
  - Automated daily backups (configure with cron)
  - Test backup restoration regularly
  - Store backups in separate location
  - Encrypt backup files
  ```bash
  # Add to crontab
  0 2 * * * cd /path/to/DachsHaus && ./scripts/backup.sh
  ```

- [ ] **Set up monitoring and alerting**
  - Failed login attempts (>10 per hour)
  - Database connection failures
  - High CPU/memory usage
  - Container restarts (>3 per hour)
  - Kafka consumer lag (>1000 messages)

### Kubernetes Security (if deploying to K8s)

- [ ] **Enable RBAC**
  - Already configured in k8s manifests
  - Review service account permissions
  - Principle of least privilege

- [ ] **Enable Pod Security Standards**
  - Enforce restricted pod security
  - Disable privileged containers
  - Configure security contexts

- [ ] **Enable network policies**
  - Already configured in k8s/base/network-policies.yaml
  - Test policy enforcement
  - Deny all by default

- [ ] **Enable Istio mTLS**
  - Already configured in k8s/base/istio/
  - Verify with `istioctl authn tls-check`
  - Enforce STRICT mode

- [ ] **Secure Kubernetes API**
  - Use private GKE cluster
  - Enable Binary Authorization
  - Restrict API access with firewall rules

### Compliance & Privacy

- [ ] **GDPR compliance** (if applicable)
  - Implement data subject access requests (DSAR)
  - Right to be forgotten (data deletion)
  - Data portability
  - Privacy policy and cookie consent

- [ ] **PCI DSS compliance** (if handling payments)

  - Use payment gateway (Stripe, PayPal, etc.)
  - Implement tokenization

- [ ] **Document security policies**

## 🚨 Incident Response

If you suspect a security breach:

1. **Contain**
   - Immediately rotate all secrets
   - Block suspicious IPs
   - Take affected services offline if necessary

2. **Investigate**
   - Review logs: `docker-compose logs | grep "suspicious-pattern"`
   - Check database for unauthorized changes
   - Identify scope of breach

3. **Remediate**
   - Patch vulnerabilities
   - Remove malicious code
   - Restore from clean backup if necessary

4. **Notify**
   - Inform affected users (within 72 hours for GDPR)
   - Report to relevant authorities
   - Document the incident

5. **Learn**
   - Post-mortem analysis
   - Update security procedures
   - Implement preventive measures

## ✅ Quick Security Verification

Run this quick check after installation:

```bash
# 1. Check file permissions
ls -la .env CREDENTIALS.txt keys/*.pem

# 2. Verify secrets are not in git
git ls-files | grep -E '\.env|CREDENTIALS|\.pem'
# Should return nothing

# 3. Check for known vulnerabilities
pnpm audit --production

# 4. Verify services are using secure defaults
docker-compose config | grep -i password
# Should not show plaintext passwords

# 5. Test authentication
curl -X POST http://localhost:4000/graphql \
  -H "Content-Type: application/json" \
  -d '{"query":"mutation { login(email:\"admin@dachshaus.local\", password:\"admin123\") { accessToken } }"}'
```
